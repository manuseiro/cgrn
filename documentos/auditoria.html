CRÍTICO Rate Limiting prometido no README não está implementado
api/proxy.php
O README afirma: "Rate limiting: 60 requisições por minuto por sessão". O código do proxy não contém nenhuma linha sequer de rate limiting. Qualquer agente pode fazer requisições ilimitadas, usando o proxy como amplificador contra as APIs governamentais (FUNAI, IBAMA, INPE etc.) ou para gerar tráfego abusivo a partir do seu servidor.
✅ Correção sugerida — adicionar no início do proxy.php
// Rate limiting: 60 req/min por IP
session_start();
$ip = $_SERVER['REMOTE_ADDR'];
$key = 'rl_' . md5($ip);
$now = time();
$window = 60;
$limit  = 60;

if (!isset($_SESSION[$key])) {
    $_SESSION[$key] = ['count' => 0, 'start' => $now];
}
if (($now - $_SESSION[$key]['start']) > $window) {
    $_SESSION[$key] = ['count' => 0, 'start' => $now];
}
$_SESSION[$key]['count']++;
if ($_SESSION[$key]['count'] > $limit) {
    http_response_code(429);
    header('Retry-After: 60');
    echo json_encode(['error' => 'Rate limit excedido. Tente em 1 minuto.']);
    exit;
}
CRÍTICO Header Injection via Content-Type não sanitizado
api/proxy.php · linha 118
O Content-Type retornado pela API remota é repassado diretamente ao header() sem nenhuma sanitização. Um servidor remoto comprometido (ou que responda com dados inesperados) poderia injetar quebras de linha (\r\n) e adicionar headers HTTP arbitrários na resposta — técnica conhecida como HTTP Response Splitting.
❌ Código atual
header("Content-Type: $contentType");
✅ Correção
// Remove qualquer caractere de controle (CRLF injection)
$safeContentType = preg_replace('/[\r\n]/', '', $contentType);
header("Content-Type: $safeContentType");
ALTO SSRF via redirecionamentos ilimitados (FOLLOWLOCATION sem MAXREDIRS)
api/proxy.php · linha 87
CURLOPT_FOLLOWLOCATION = true sem CURLOPT_MAXREDIRS permite redirecionamentos em cadeia potencialmente infinitos. Se uma API autorizada redirecionar para um host interno (e.g., http://169.254.169.254/ — AWS metadata), ou para um protocolo diferente como file://, o cURL seguirá sem restrição. Isso abre vetor de SSRF (Server-Side Request Forgery).
❌ Código atual
curl_setopt($ch, CURLOPT_FOLLOWLOCATION, true);
✅ Correção
curl_setopt($ch, CURLOPT_FOLLOWLOCATION, true);
curl_setopt($ch, CURLOPT_MAXREDIRS, 3);
curl_setopt($ch, CURLOPT_PROTOCOLS, CURLPROTO_HTTPS | CURLPROTO_HTTP);
curl_setopt($ch, CURLOPT_REDIR_PROTOCOLS, CURLPROTO_HTTPS); // Só HTTPS em redirect
MÉDIO XSS Stored: project.name inserido em innerHTML sem escape
js/services/persistence.js · linhas 37, 72, 115
O nome do projeto digitado pelo usuário é salvo no localStorage e depois injetado diretamente em innerHTML em múltiplos pontos. Se um usuário (ou arquivo .cgrn importado de terceiro) contiver um nome como <img src=x onerror=alert(1)>, ele será executado como XSS na próxima carga do projeto.
❌ Código atual (3 ocorrências)
showProjectMessage(`Projeto "<strong>${project.name}</strong>" salvo`, ...);
el.innerHTML = `<div class="fw-semibold">${project.name}</div>`;
✅ Correção — adicionar função de escape
// Adicionar em ui.js ou utils
export function escapeHtml(str) {
  const d = document.createElement('div');
  d.textContent = str;
  return d.innerHTML;
}

// Em persistence.js:
showProjectMessage(`Projeto "<strong>${escapeHtml(project.name)}</strong>" salvo`, ...);
el.innerHTML = `<div class="fw-semibold">${escapeHtml(project.name)}</div>`;

BUG Cache hit em validateCoordinates retorna dados incompletos
js/services/validation.js · linhas 302–310
Quando há cache hit, a função retorna overlapFeatures: undefined e warnings: [] (sem os avisos originais). O código que chama a função (em main.js) pode tentar iterar result.overlapFeatures e quebrar silenciosamente. Além disso, os avisos de duplicatas/interseção gerados na primeira validação são perdidos nas chamadas subsequentes com cache.
❌ Código atual
if (state.cache.has(cacheKey)) {
  return {
    valid: true, errors: [],
    warnings: [],            // ← avisos perdidos
    data: state.cache.get(cacheKey),
    fromCache: true          // ← overlapFeatures ausente
  };
}
✅ Correção — cachear o resultado completo
// Cachear o resultado inteiro, não só allData
state.cache.set(cacheKey, { data: allData, warnings: allWarnings, overlapFeatures });

if (state.cache.has(cacheKey)) {
  const cached = state.cache.get(cacheKey);
  return { valid: true, errors: [], fromCache: true, ...cached };
}
BUG sicor.js: WKT MultiPolygon parseado como Polygon com múltiplos anéis
js/services/sicor.js · função parseWKT()
Quando o WKT é um MULTIPOLYGON com mais de um polígono, o código coleta todos os rings e cria turf.polygon(rings, props). Para o Turf.js, passar múltiplos rings em turf.polygon() cria um polígono com buracos (holes), não um MultiPolygon. O resultado geométrico é incorreto e pode gerar falsos negativos na verificação de financiamentos SICOR.
❌ Código atual
return turf.polygon(rings, properties); // rings[1+] viram "holes"
✅ Correção
if (rings.length === 1) {
  return turf.polygon(rings, properties);
} else {
  // Cada ring é um polígono independente no MultiPolygon
  return turf.multiPolygon(rings.map(r => [r]), properties);
}
BUG CORS preflight (OPTIONS) não tratado — APIs podem bloquear requisições
api/proxy.php
O proxy não responde ao método HTTP OPTIONS (preflight CORS). Em navegadores modernos, qualquer requisição cross-origin com headers customizados ou métodos não-simples dispara um preflight. Sem tratamento do OPTIONS, o proxy retorna erro ou código inesperado, podendo bloquear requisições legítimas em produção onde o CORS é restrito.
✅ Correção — adicionar logo após os headers CORS
header('Access-Control-Allow-Methods: GET, OPTIONS');
header('Access-Control-Allow-Headers: Content-Type, X-Requested-With');

if ($_SERVER['REQUEST_METHOD'] === 'OPTIONS') {
    http_response_code(204);
    exit;
}
LÓGICA Discrepância entre AREA_MAX_HA no código (15.000 ha) e README (50.000 ha)
js/utils/config.js · VALIDATION.AREA_MAX_HA
O README documenta limite máximo de 50.000 ha por gleba. O código em config.js usa 15.000 ha e conformidade.js usa AREA_MAX_HA para gerar alerta. Essa inconsistência pode reprovar operações legítimas de grandes imóveis rurais ou dar conformidade errada ao usuário. Definir o valor correto conforme a Resolução CMN vigente.
Verificar qual é o limite real da Res. CMN 5.081/2023 para a modalidade geral de crédito rural e alinhar código, config.js e documentação.

Melhorias Sugeridas

6 sugestões
MELHORIA Substituir hash djb2 por chave de cache mais robusta
js/services/validation.js · função djb2()
O hash djb2 de 32 bits gera colisões teoricamente improváveis, mas para uma aplicação geoespacial com coordenadas muito parecidas (diferindo apenas na última casa decimal), a probabilidade aumenta. Uma colisão faria a validação retornar dados de outra gleba silenciosamente. Solução mais simples: usar as primeiras 200 caracteres do texto como chave (textos de glebas são únicos na prática). 
// Simples, sem risco de colisão para textos de gleba
const cacheKey = `${text.slice(0, 200)}_${validarPontos ? '1' : '0'}`;
MELHORIA Validação de área mínima/máxima movida para validateSingleGleba
js/services/validation.js
Atualmente, a validação de área (min/max hectares) só ocorre em conformidade.js → checkArea(), que é assíncrona e opcional. Glebas fora dos limites de área passam pela validação e chegam ao mapa antes de serem reprovadas. Isso cria inconsistência de UX. Adicionar a verificação de área em validateSingleGleba como erro bloqueante imediato. 
// Em validateSingleGleba(), após calcular area:
const area = turf.area(poly) / 10000;
if (area < CONFIG.VALIDATION.AREA_MIN_HA || area > CONFIG.VALIDATION.AREA_MAX_HA) {
  return { errors: [`Gleba ${glebaId}: área ${area.toFixed(2)} ha fora dos limites.`], glebaData: null };
}
MELHORIA Cache de validação sem TTL — dados ficam obsoletos entre sessões
js/utils/state.js / js/services/validation.js
O state.cache (Map em memória) nunca é limpo durante a sessão. Se o usuário recarregar novas camadas (TI, IBAMA) e revalidar as mesmas coordenadas, o cache retorna o resultado antigo sem as novas verificações. Adicionar timestamp ao cache ou invalidar ao carregar novas camadas. 
// Alternativa simples: limitar tamanho do cache
const MAX_CACHE = 20;
if (state.cache.size >= MAX_CACHE) {
  state.cache.delete(state.cache.keys().next().value); // remove o mais antigo
}
state.cache.set(cacheKey, resultado);
MELHORIA Adicionar .htaccess protegendo o diretório api/cache/
api/cache/ (diretório público)
O diretório api/cache/ armazena respostas JSON de APIs governamentais e é acessível diretamente via URL (ex: /api/cache/abc123.json). Qualquer pessoa pode inspecionar o cache e inferir quais URLs foram consultadas. Adicionar um .htaccess no diretório. 
# api/cache/.htaccess
Order deny,allow
Deny from all
MELHORIA Detecção de header CSV em upload.js é frágil
js/services/upload.js · função normalizeFlatFile()
A detecção de linha de cabeçalho usa a regex /gleba|ponto|lat|lon/i. Arquivos com cabeçalhos em inglês (ID,order,latitude,longitude) passam a regex mas "latitude" seria detectado — porém "order" e "ID" não ativam o filtro individualmente. O primeiro campo "ID" (não numérico) já é suficiente: qualquer linha onde parts[0] não seja numérico deve ser descartada. 
// Substitua o filtro de header por:
if (parts.some(v => isNaN(Number(v)))) return null;
MELHORIA Adicionar X-Proxy-Error-Msg apenas em modo dev
api/proxy.php · linha 138
O header X-Proxy-Error-Msg expõe mensagens de erro internas da biblioteca cURL (endereços IP internos, nomes de host, detalhes de SSL) para qualquer cliente em produção. Isso facilita a enumeração de recursos internos do servidor. Restringir ao modo dev. 
header('X-Proxy-Error-Msg: ' . ($error ?: "..."));

// Corrigido:
if ($isDev) {
    header('X-Proxy-Error-Msg: ' . ($error ?: "HTTP $httpCode"));
}

Pontos Fortes do Projeto

Arquitetura limpa: Uso correto de ES Modules nativos, separação clara entre services/utils/components, sem bundler desnecessário.
Índice espacial com bbox: O pré-filtro por bounding box antes de booleanPointInPolygon é a abordagem correta para performance com milhares de features.
Fallback robusto: Todas as camadas externas têm fallback para dados locais. O sistema funciona mesmo sem conexão com as APIs governamentais.
Modelagem BACEN/SICOR correta: Os status (ok/alerta/bloqueio/pendente) e a referência às resoluções legais estão bem documentados e estruturados.
Proxy com whitelist exata: O uso de in_array() com comparação exata (não substring) evita bypass por subdomain (evil.car.gov.br é corretamente rejeitado).
Cache Stale-While-Revalidate no proxy: Retornar dados expirados enquanto busca novos é uma boa prática para resiliência com APIs instáveis.
