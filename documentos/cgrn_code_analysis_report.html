// Remover o bloco das linhas 148-151 (duplicado do 175-178) // Manter apenas o listener via el.mostrarTI em bindEvents()

A cada busca bem-sucedida, btnImportarCAR e btnVerNoMapaCAR recebem um novo addEventListener sem remoção do anterior. Depois de 3 buscas, importar clica 3 vezes com dados de buscas antigas.
// Antes de setar o innerHTML, guarde a referência antiga e use cloneNode // OU use um delegado no container do resultDiv apenas uma vez
🟡
clearGlebas() não limpa validationMarkerLayers
Em state.js, clearGlebas zera polygonLayers, markerLayers e centroidLayers, mas esquece validationMarkerLayers. Os marcadores de erro (pontos duplicados / autointerseções) ficam no mapa após limpar.

bboxIntersects definida localmente em validation.js
A função já existe exportada em geo.js e é importada corretamente por ibama.js, icmbio.js e terras_indigenas.js. Em validation.js ela está redefinida como cópia local (linha 42), quebrando o princípio DRY e o propósito do módulo geo.js.
// validation.js — linha 1 import { bboxIntersects } from '../utils/geo.js'; // ↓ e remover a função local das linhas 41-44

Warnings de validação não exibidos em processAndRender
validateCoordinates retorna result.warnings (duplicatas, interseções), mas em processAndRender esses warnings são ignorados — só aparecem ao usar "Validar" pelo botão dedicado. Usuários que clicam "Adicionar Gleba" não veem os avisos.
// Em processAndRender(), após glebas = result.data: if (result.warnings?.length) { showToast(result.warnings, 'warning', 8000); }
🟡
Discrepância: AREA_MAX_HA: 15000 vs README (50.000 ha)
O README documenta o limite máximo como 50.000 ha, mas config.js define AREA_MAX_HA: 15000. Qualquer gleba entre 15.001 e 50.000 ha recebe alerta indevido. Deve-se alinhar os dois ou tornar o valor configurável.
Segurança atenção
🔴
XSS via e.message em innerHTML
Em main.js (linha 735): resultDiv.innerHTML = `...Erro na consulta: ${e.message}...`. Se a API retornar uma mensagem com tags HTML ou a exceção contiver dados externos, há risco de Cross-Site Scripting.
resultDiv.textContent = `Erro na consulta: ${e.message}`; // OU use innerText — nunca innerHTML com dados externos
🟡
Domínio github.com na whitelist do proxy
O proxy.php permite qualquer URL em github.com. Isso abre SSRF para endpoints como api.github.com, repositórios privados, ou outros sub-recursos. Substitua por manuseiro.github.io (o destino real).
// proxy.php — whitelist $allowedDomains = [ 'manuseiro.github.io', // ← específico, não 'github.com' 'geoservices.icmbio.gov.br', // ...
🟡
Rate limiting documentado, mas não implementado no proxy
O README descreve "60 req/min por sessão" como feature de segurança, mas o proxy.php não tem nenhuma lógica de rate limiting. Qualquer script pode inundar os servidores governamentais via proxy sem limite.
// Implementar com session + contador + timestamp em $_SESSION session_start(); $key = 'rl_' . md5($targetUrl); $now = time(); // verificar $_SESSION[$key] e bloquear se > 60/min

Validações Geoespaciais — Auto-interseção, Duplicatas e Sobreposição lacuna crítica
✅
Auto-interseções — implementado e visível no mapa
Detectadas com turf.kinks() em validateSingleGleba. Ao ativar "Validar Pontos", marcadores âmbar são renderizados exatamente no ponto de cruzamento via renderValidationMarkers(). Também verificado em checkGeometria no fluxo BACEN/SICOR. Funciona corretamente.
✅
Pontos duplicados consecutivos — implementado e visível
Detectados em validateSingleGleba com comparação de coordenadas exatas. Ao ativar "Validar Pontos", marcadores vermelhos grandes (radius 9) são renderizados sobre o ponto duplicado com popup informativo. Bem executado.
❌
Sobreposição entre glebas (gleba vs gleba) — NÃO implementado
Quando o usuário insere múltiplas glebas, o sistema não verifica se uma gleba sobrepõe outra. A conformidade BACEN/SICOR verifica sobreposição com CAR, TI, UC, IBAMA — mas nunca entre as próprias glebas do projeto. Uma área inserida duas vezes, ou duas glebas sobrepostas, passa sem aviso.
// Adicionar em validateCoordinates(), após construir allData: for (let i = 0; i < allData.length; i++) { for (let j = i + 1; j < allData.length; j++) { try { const inter = turf.intersect(allData[i].turfPolygon, allData[j].turfPolygon); if (inter) { const areaHa = turf.area(inter) / 10000; allWarnings.push( `Glebas ${allData[i].glebaId} e ${allData[j].glebaId} ` + `se sobrepõem em ~${areaHa.toFixed(2)} ha.` ); } } catch(_) {} } }
⚠️
Pontos não-consecutivos duplicados — não detectados
A verificação atual só compara pontos consecutivos (prev[0] === curr[0]). Um polígono com ponto repetido em posições 2 e 8 (não adjacentes) não é detectado. Complementar com uma verificação de Set de coordenadas únicas.
Arquitetura e Organização boa base
🟡
main.js com 753 linhas — candidato a divisão
O arquivo concentra orquestração de eventos, renderização do modal de conformidade (showConformidadeDetalhe, buildCARPanel) e busca CAR. Essas últimas duas funções somam ~280 linhas e pertencem a um módulo dedicado como components/conformidade-ui.js.
🟡
Imports no meio de arquivo (terras_indigenas.js linha 228)
O import de bboxIntersects está no final do arquivo, após todo o código. Em ES Modules, funciona por hoisting, mas viola a convenção e prejudica a legibilidade — imports devem ser sempre no topo.
🟡
UC_PROTECAO_INTEGRAL exportado no final do arquivo
Em camadas_externas.js, as constantes são declaradas no início sem export e reexportadas no final (linha 584) via export { ... }. Prefira declarar com export const no ponto de definição para clareza imediata.
✅
Separação de responsabilidades bem aplicada
Divisão em components/, services/ e utils/ é clara. validation.js sem efeitos DOM, dom.js isolando seletores, config.js com Object.freeze() — padrões sólidos.
✅
Cache DJB2 em validation.js — bem pensado
Usar hash para chave de cache em vez do texto bruto é inteligente. A combinação djb2(text) + validarPontos cobre a variação de contexto. Ponto de atenção: o cache não tem limite de tamanho — em sessões longas pode crescer indefinidamente.
✅
Índice espacial por bbox — excelente para performance
O pré-filtro bboxIntersects antes de turf.booleanIntersects reduz as comparações geométricas de O(N) para ~O(k), onde k é pequeno. Padrão correto para dados geoespaciais com muitas features (IBAMA ~11 MB).
Performance e Escalabilidade oportunidades
🟡
Cache de validação sem limite de tamanho
O state.cache (Map) cresce indefinidamente. Em sessões com muitas edições de coordenadas, pode consumir memória significativa. Implementar LRU simples com limite de 50–100 entradas.
// Em validateCoordinates(), após cache.set(): if (state.cache.size > 100) { state.cache.delete(state.cache.keys().next().value); }
🟡
Math.max(...glebas.map(...)) com spread em arrays grandes
Em main.js linhas 204 e 710: Math.max(...state.glebas.map(g => g.glebaId)). O spread operator pode causar stack overflow com centenas de glebas. Use Math.max(...) com reduce.
const nextGid = state.glebas.reduce( (max, g) => g.glebaId > max ? g.glebaId : max, 0 ) + 1;
🟡
Import dinâmico de map.js dentro de fluxo crítico
Em processAndRender e em listeners de eventos: import('./components/map.js').then(...). O módulo já está carregado (é importado estaticamente no topo), o import dinâmico é desnecessário e introduz uma microtask de latência.
// Substituir por import estático no topo do arquivo: import { renderValidationMarkers, renderCARLayer } from './components/map.js';
🟡
EXAMPLE_COORDS com precisão inconsistente
O exemplo usa 4 casas decimais (-6.2410) enquanto COORD_PRECISION: 8. O usuário que clica "Inserir Exemplo" e depois valida pode receber avisos de precisão. Padronizar o exemplo com 8 casas decimais.
🟡
CONFIG.DEBUG: false — logs sem efeito em produção
Todas as chamadas log() e warn() em 18+ arquivos são no-ops em produção. O padrão é correto, mas importar e chamar essas funções em todo módulo adiciona overhead mínimo porém desnecessário. Considere globalThis.__DEV__ removido pelo bundler no build.
